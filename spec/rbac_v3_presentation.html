Architecture새로운 데이터 모델

현재 (v1/v2)

g_roles ├── g_role_org_permissions │ └── permission ├── g_role_project_permissions │ └── projectId + permission └── g_role_environment_permissions └── environmentId + permission g_user_roles (userId → roleId) g_group_roles (groupId → roleId)

5개 테이블, 범위가 역할 안에 혼재

→

제안 (v3)

g_roles └── g_role_permissions └── permission (범위 없음!) g_role_bindings ├── userId / groupId ├── roleId ├── scopeType (system|org|project|env) └── scopeId

2개 테이블, 역할과 범위 완전 분리

Roles역할 = 순수 권한 집합

Developer

features:* segments:read client_versions:update change_requests:create

Viewer

*:read

System Admin

*:*
  • 역할에 projectId / environmentId 없음
  • "이 역할이 어디에서 적용되는지"는 바인딩이 결정
  • 와일드카드: *:*, features:*, *:read

Binding역할 바인딩 = 스코프별 할당

scopeType scopeId 의미 우선순위
system 'SYSTEM' 전체 시스템 (System Admin) 낮음 (최상위 fallback)
org orgId 조직 내 모든 프로젝트에 기본 적용 ↑
project projectId 특정 프로젝트에서 오버라이드 ↑
environment environmentId 특정 환경에서 오버라이드 높음 (최우선)
핵심 규칙: 가장 구체적인 바인딩이 이김 (Override)

env 바인딩 > project 바인딩 > org 바인딩 > system 바인딩

바인딩 관리 (신규)

사용자: user@example.com 🏢 조직 기본: Developer 📁 Project GameA: (기본값) 📁 Project GameB: Viewer ✏️ 오버라이드 🌍 GameA/prod: ReadOnly ✏️ 오버라이드 [+ 바인딩 추가]

정리

  • 멤버십 = 접근 게이트 — 정합성 문제 해결
  • 역할 = 순수 권한 — 범위 혼재 제거
  • 바인딩 = 스코프별 할당 — 오버라이드 지원
  • 테이블 순감 3개 — 복잡도 감소
  • 4단계 스코프 — system > org > project > env
