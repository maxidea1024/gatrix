# Gatrix 이미지 메모리 최적화 — 개선 보고서

## 개요

Gatrix SDK의 이미지 로딩/디스플레이 시스템을 전면 리팩토링하여 **텍스처 공유**, **백그라운드 디코딩**, **네트워크 캐싱**을 구현했습니다.

---

## 핵심 변경 사항

### 1. 텍스처 공유 (핵심 개선)

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| **텍스처 소유** | 위젯마다 개별 텍스처 생성 | URL 당 1세트 공유 |
| **동일 URL 5개 위젯** | 텍스처 5세트 (N프레임 × 5) | **1세트** (N프레임 × 1) |
| **VRAM 절감** | - | 동일 URL 다중 사용 시 **80~95% 감소** |

> **변경 이유**: 기존에는 각 UGatrixWebImage / UGatrixBannerWidget이 GIF 프레임별로 UTexture2DDynamic을 개별 생성했습니다. 같은 URL 이미지를 N곳에서 사용하면 N배의 VRAM이 낭비되었습니다.
>
> **개선 내용**: UGatrixImageManager가 URL당 하나의 프레임 텍스처 세트를 소유하고, 모든 소비자가 이를 공유합니다. Manager의 Tick에서 프레임 인덱스를 증가시키면 소비자가 GetDisplayTexture()로 현재 프레임 텍스처를 참조합니다.
>
> ⚠️ **주의**: 단일 URL을 하나의 위젯에서만 사용하는 경우, VRAM 사용량은 기존과 동일합니다 (프레임당 1개 텍스처). 절감 효과는 동일 URL의 다중 사용 시에만 발생합니다.

---

### 2. 완전 비동기 디코딩

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| **디코딩 위치** | 게임 스레드에서 실행 가능 | **무조건 백그라운드 스레드** |
| **게임 스레드 비용** | GIF 디코딩 50-100ms 블로킹 | **0ms** (인덱스 교체만) |
| **프레임 전환 비용** | RGBA → GPU 업로드 (2MB/프레임) | 사전 생성 텍스처 포인터 교체 |

> **변경 이유**: 기존 "슬라이딩 윈도우" 방식은 윈도우 밖 프레임이 필요할 때 게임 스레드에서 전체 GIF를 재디코딩하여 심각한 프레임 드랍(1~5 FPS)이 발생했습니다.
>
> **개선 내용**: 비동기 태스크에서 전체 프레임을 한 번에 디코딩 → 게임 스레드에서 CreateTextureFromRGBA로 텍스처 생성 → 이후 재디코딩 없음. Tick에서는 정수 인덱스만 증가시킵니다.

---

### 3. ETag 기반 조건부 요청 (대역폭 최적화)

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| **배너 목록 API** | 매번 전체 데이터 전송 | If-None-Match → 304 시 0바이트 |
| **2회차 이후 요청** | 동일한 JSON 재전송 | **304 Not Modified** (수 바이트) |
| **오프라인 지원** | 불가 | 디스크 캐시에서 로드 |

> **변경 이유**: 배너 목록은 자주 변경되지 않지만 매번 전체 JSON을 전송받아 불필요한 대역폭과 파싱 비용이 발생했습니다.
>
> **개선 내용**: 서버의 ETag 응답 헤더를 저장하고, 후속 요청 시 `If-None-Match` 헤더로 전송합니다. 서버가 304를 반환하면 기존 캐시를 그대로 사용합니다.

---

### 4. 배너 디스크 캐싱 + 이미지 프리페치

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| **앱 시작 시** | 네트워크 응답 대기 후 표시 | 디스크 캐시에서 즉시 표시 |
| **이미지 로딩** | 프레임 전환 시 다운로드 | 사전 다운로드 완료 상태 |
| **네트워크 끊김** | 배너 표시 실패 | 캐시에서 정상 표시 |

> **변경 이유**: 앱 시작 시 배너를 표시하려면 네트워크 응답을 기다려야 했고, 이미지도 순차적으로 다운로드되어 첫 프레임 표시까지 지연이 발생했습니다.
>
> **개선 내용**: (1) 배너 JSON을 디스크에 캐싱하여 초기 로딩 즉시 표시, (2) 배너 목록 수신 후 모든 이미지 URL을 자동으로 프리페치하여 디스크에 저장합니다.

---

### 5. TTL 기반 지연 제거

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| **참조 해제 시** | 즉시 제거 | 30초 유지 후 제거 (설정 가능) |
| **프레임 전환 시** | 해제 → 재생성 (낭비) | 기존 소스 재활용 |

> **변경 이유**: 배너 프레임 전환 시 이전 프레임의 이미지 소스를 즉시 제거하면, 다시 해당 프레임으로 돌아갈 때 재다운로드+재디코딩이 필요합니다.
>
> **개선 내용**: 참조 카운트가 0이 되어도 일정 시간(기본 30초) 동안 소스를 유지합니다. 이 기간 내에 다시 참조되면 기존 소스를 재활용합니다.

---

### 6. 콘솔 진단 명령어

| 명령어 | 기능 |
|--------|------|
| `Gatrix.ImageStats` | 전체 소스별 상세 통계 (메모리, 상태, 소비자 수 등) |
| `Gatrix.ImageStats.Summary` | 한줄 요약 (소스 수, RAM, VRAM) |
| `Gatrix.ImageConfig` | 현재 설정값 표시 |
| `Gatrix.ImageEvict` | 즉시 제거 검사 실행 |
| `Gatrix.ImageReleaseAll` | 모든 이미지 소스 강제 해제 |

## 설정 (Config/Uwo.ini)

```ini
[Gatrix]
; 이미지 소스 최대 수 (기본: 64)
ImageMaxSources=64

; 참조 해제 후 제거 대기 시간 (초, 기본: 30)
ImageEvictionGracePeriod=30.0

; 제거 검사 주기 (초, 기본: 5)
ImageEvictionCheckInterval=5.0

; 동시 다운로드 수 (기본: 4)
ImageMaxConcurrentDownloads=4
```

## 메모리 비교 시나리오

### 시나리오 A: 동일 URL 이미지를 5곳에서 동시 표시

| 항목 | 개선 전 | 개선 후 | 절감률 |
|------|---------|---------|--------|
| 다운로드 횟수 | 5회 | **1회** | 80% |
| GPU 텍스처 수 | 5세트 | **1세트** | 80% |
| VRAM | ×5 | ×1 | **80%** |
| 디코딩 횟수 | 5회 | **1회** | 80% |

### 시나리오 B: 서로 다른 URL 10개 배너 (각 70프레임 200×200 GIF)

| 항목 | 개선 전 | 개선 후 | 비고 |
|------|---------|---------|------|
| GPU 텍스처 수 | 700개 | **700개** | URL이 모두 다르면 동일 |
| VRAM | 112MB | **112MB** | 공유 대상 없음 → 동일 |
| 게임 스레드 비용/tick | 텍스처 포인터 교체 | **인덱스 증가만** | 동일 수준 |
| 디코딩 위치 | 게임 스레드 가능 | **100% 백그라운드** | FPS 안정화 |

> **핵심**: VRAM 절감은 동일 URL의 공유에서 발생합니다. 서로 다른 URL의 고유 이미지는 기존과 동일한 VRAM을 사용합니다. 주된 개선은 **공유**, **비동기 디코딩**, **네트워크 캐싱**입니다.

## 아키텍처 변경 Trade-off

| 접근 방식 | VRAM | CPU/Tick | 끊김 |
|-----------|------|----------|------|
| 프레임당 텍스처 (현재) | 높음 | **제로** (인덱스 교체) | 없음 ✅ |
| 단일 텍스처 + 매 틱 GPU 업로드 | 낮음 | 높음 (RGBA 복사+업로드) | 가능 ❌ |
| 슬라이딩 윈도우 재디코딩 | 중간 | 매우 높음 (풀 디코딩) | 심각 ❌ |

> 프레임당 텍스처 방식은 VRAM을 더 사용하지만, 게임 스레드 비용이 0이고 끊김이 없어 실사용에 가장 적합합니다.
